# 챌린지 매니저 플랫폼 프론트엔드 디자인 기획서

## 1. 문서 목적

이 문서는 `챌린지_매니저_플랫폼_PRD_구현착수용.md`를 바탕으로 1차 구현에 필요한 프론트엔드 화면 구조, 사용자 경험, 디자인 시스템, 컴포넌트, 상태 표현, 반응형 기준을 정의한다.

목표는 예쁜 소개 페이지가 아니라 실제 챌린지 운영자가 매일 쓰고, 참가자가 모바일에서 빠르게 인증을 제출할 수 있는 실용적인 웹앱을 설계하는 것이다.

## 2. 제품 UX 방향

## 2.1 제품 성격

챌린지 매니저는 운영형 SaaS에 가깝다.

따라서 첫 화면은 마케팅 랜딩이 아니라 로그인 후 바로 사용할 수 있는 대시보드 또는 챌린지 목록이어야 한다.

주요 사용 맥락:

- 관리자는 데스크톱에서 여러 참가자, 인증, 상태를 빠르게 확인한다.
- 참가자는 모바일에서 챌린지 확인과 링크 인증 제출을 빠르게 끝낸다.
- 운영자는 고객명, 청구 상태, 성공자 CSV를 통해 수익화 검증까지 관리한다.

## 2.2 디자인 톤

추천 톤:

- 차분한 운영툴
- 명확한 상태 구분
- 높은 정보 밀도
- 신뢰감 있는 B2B SaaS
- 모바일에서는 간결한 작업 중심 UI

피해야 할 톤:

- 과한 랜딩 페이지 스타일
- 대형 히어로 섹션 중심 구성
- 카드가 너무 많은 장식형 대시보드
- 보라색 그라데이션 중심의 흔한 SaaS 느낌
- 챌린지라는 이유로 지나치게 게임화된 화면

## 2.3 핵심 디자인 원칙

- 상태가 먼저 보여야 한다.
- 관리자는 목록에서 바로 판단할 수 있어야 한다.
- 참가자는 3번 이하의 탭으로 인증을 제출할 수 있어야 한다.
- 금전, 성공, 반려 상태는 애매한 색이나 문구를 쓰지 않는다.
- 관리자와 참가자의 화면 밀도는 다르게 설계한다.
- 모바일 참가자 화면은 입력 부담을 최소화한다.
- CSV, 감사 로그, 청구 상태 같은 운영 기능은 숨기지 않고 명확히 배치한다.

## 3. 정보 구조

## 3.1 전체 라우트 구조

```text
/
/login
/register

/challenges
/challenges/:id
/my
/my/challenges/:id
/my/challenges/:id/submit
/my/submissions

/admin
/admin/challenges
/admin/challenges/new
/admin/challenges/:id/edit
/admin/challenges/:id/participants
/admin/submissions
/admin/challenges/:id/ranking
/admin/audit-logs
```

## 3.2 권한별 시작 화면

관리자 로그인 후:

- `/admin`
- 운영 현황 대시보드

참가자 로그인 후:

- `/challenges`
- 모집 중/진행 중 챌린지 목록

비로그인 사용자:

- `/login`
- 회원가입 링크 제공
- 챌린지 상세는 공개 조회 가능 여부를 개발 시 결정할 수 있으나, 1차 구현에서는 로그인 후 조회를 기본으로 권장한다.

## 4. 레이아웃 전략

## 4.1 관리자 레이아웃

관리자 화면은 데스크톱 우선으로 설계한다.

구성:

- 좌측 사이드바
- 상단 바
- 본문 작업 영역
- 테이블 중심 화면
- 필요한 경우 우측 상세 패널 또는 모달

사이드바 메뉴:

- 대시보드
- 챌린지
- 인증 심사
- 감사 로그

상단 바:

- 현재 페이지 제목
- 주요 액션 버튼
- 로그인 사용자명
- 로그아웃

본문 폭:

- 데스크톱에서는 전체 폭을 적극 활용한다.
- 테이블은 가로 스크롤보다 컬럼 우선순위를 조정한다.
- 중요한 액션은 우측 상단 또는 행 단위 액션으로 배치한다.

## 4.2 참가자 레이아웃

참가자 화면은 모바일 우선으로 설계한다.

구성:

- 상단 간단 헤더
- 하단 탭 또는 상단 탭
- 카드형 챌린지 목록
- 인증 제출 폼
- 상태 배지 중심의 내역 화면

참가자 주요 메뉴:

- 챌린지
- 내 챌린지
- 인증 내역

모바일 기준:

- 한 화면에서 주요 CTA가 명확해야 한다.
- 인증 제출 화면은 스크롤이 짧아야 한다.
- 링크 입력 후 제출까지 흐름을 방해하는 부가 정보는 최소화한다.

## 5. 화면별 디자인 기획

## 5.1 로그인 화면

목적:

- 관리자와 참가자가 같은 로그인 화면을 사용한다.
- 로그인 후 role에 따라 이동한다.

구성:

- 서비스명
- 이메일 입력
- 비밀번호 입력
- 로그인 버튼
- 참가자 회원가입 링크
- 오류 메시지 영역

디자인:

- 좌우 분할 레이아웃은 피한다.
- 중앙 정렬의 좁은 폼을 사용한다.
- 배경은 밝은 회색 또는 아주 연한 톤으로 두고 폼은 과하게 장식하지 않는다.

상태:

- 기본
- 입력 오류
- 로그인 실패
- 로딩

## 5.2 회원가입 화면

목적:

- 참가자 계정을 만든다.
- 관리자 계정 생성은 별도 seed 또는 운영자 등록으로 처리한다.

필드:

- 이름
- 이메일
- 비밀번호
- 휴대폰 번호

UX:

- 비밀번호 조건은 입력 하단에 작게 표시한다.
- 이메일 중복 오류는 해당 필드 아래에 표시한다.
- 가입 완료 후 로그인 화면으로 이동한다.

## 5.3 참가자 챌린지 목록

목적:

- 참가자가 참여 가능한 챌린지를 빠르게 찾는다.

화면 구성:

- 상단 헤더
- 상태 탭: 모집 중, 진행 중
- 챌린지 카드 목록

카드 표시 정보:

- 챌린지 제목
- 상태 배지
- 모집 기간
- 진행 기간
- 참가비 표시 금액
- 성공 기준
- 현재 참가자 수
- 상세 보기 버튼

카드 CTA:

- 모집 중: `상세 보기`
- 진행 중: `진행 상황 보기`

빈 상태:

- “현재 모집 중인 챌린지가 없습니다.”

## 5.4 참가자 챌린지 상세

목적:

- 참가 조건을 이해하고 신청한다.

구성:

- 챌린지 제목
- 상태 배지
- 기간 정보
- 참가비 표시 금액
- 성공 기준 요약
- 인증 안내
- 환불 안내
- 참가 신청 버튼

중요 UX:

- 참가 신청 버튼은 화면 하단 고정 영역에 둘 수 있다.
- 신청 불가 사유는 버튼 위에 짧게 표시한다.
- 이미 신청한 경우 `내 챌린지 보기` 버튼으로 전환한다.

## 5.5 내 챌린지 화면

목적:

- 참가자가 자신의 진행 상태를 확인한다.

카드 표시 정보:

- 챌린지 제목
- 참가 상태
- 결제 상태
- 승인된 회차 수
- 성공 필요 승인 수
- 현재 순위
- 성공 여부

진행률 표현:

- `승인 3 / 필요 5`
- 진행률 바 사용
- 성공 가능성은 과장하지 않고 현재 상태만 표현

CTA:

- 인증 제출
- 인증 내역 보기

## 5.6 인증 제출 화면

목적:

- 참가자가 블로그 링크 인증을 빠르게 제출한다.

필드:

- 미션 회차
- 블로그 링크
- 인증 설명

권장 UI:

- 미션 회차는 셀렉트 또는 스텝퍼
- 링크 입력은 큰 단일 입력창
- 설명은 3~5줄 텍스트 영역
- 제출 버튼은 하단 고정 또는 폼 끝에 명확히 배치

검증 메시지:

- “참가 확정 후 제출할 수 있습니다.”
- “진행 기간에만 제출할 수 있습니다.”
- “URL 형식으로 입력해주세요.”
- “인증 설명은 10자 이상 입력해주세요.”

제출 완료:

- 성공 토스트
- 인증 내역 화면으로 이동

## 5.7 인증 내역 화면

목적:

- 제출 결과와 반려 사유를 확인한다.

표시 정보:

- 회차
- 링크
- 제출 시각
- 상태
- 반려 사유
- 심사 시각

상태 표현:

- 검토 중: 회색 또는 파란색 배지
- 승인: 녹색 배지
- 반려: 빨간색 배지

반려된 항목:

- 반려 사유를 카드 안에서 바로 보여준다.
- `재제출` 버튼을 제공한다.

## 5.8 관리자 대시보드

목적:

- 오늘 처리해야 할 운영 이슈를 빠르게 확인한다.

상단 지표:

- 전체 챌린지 수
- 모집 중 챌린지 수
- 진행 중 챌린지 수
- 검토 대기 인증 수
- 결제 대기 참가자 수
- 성공자 수

하단 구성:

- 최근 검토 대기 인증
- 최근 생성 챌린지
- 청구 상태 요약

디자인:

- 카드 남발을 피한다.
- 핵심 지표는 작은 메트릭 블록으로 배치한다.
- 테이블과 리스트 중심으로 운영 효율을 높인다.

## 5.9 관리자 챌린지 목록

목적:

- 챌린지를 운영 단위로 관리한다.

테이블 컬럼:

- 제목
- 상태
- 모집 기간
- 진행 기간
- 참가자 수
- 검토 대기 인증 수
- 고객명
- 청구 상태
- 예상 청구 금액
- 액션

필터:

- 상태
- 청구 상태
- 검색어

행 액션:

- 수정
- 참가자
- 인증 심사
- 랭킹
- CSV

중요 UX:

- 청구 상태는 관리자만 볼 수 있다.
- 청구 상태 변경은 모달 또는 인라인 셀렉트로 처리한다.

## 5.10 챌린지 생성/수정 화면

목적:

- 운영자가 챌린지를 생성하고 조건을 확정한다.

섹션 구성:

1. 기본 정보
2. 기간 설정
3. 성공 기준
4. 참가 및 안내
5. 고객/청구 정보
6. 상태 설정

주요 입력:

- 제목
- 설명
- 모집 기간
- 진행 기간
- 참가비 표시 금액
- 전체 미션 수
- 성공 필요 승인 수
- 최대 참가자 수
- 인증 안내
- 환불 안내
- 고객명
- 고객 유형
- 과금 방식 메모
- 예상 청구 금액
- 청구 상태
- 상태

UX:

- 한 페이지 폼으로 시작한다.
- 섹션별 구분선을 사용한다.
- 저장 버튼은 상단 우측과 하단에 배치한다.
- `active` 이후 수정 제한 필드는 비활성화 표시한다.

## 5.11 관리자 참가자 관리

목적:

- 참가 신청자와 결제 상태를 관리한다.

테이블 컬럼:

- 이름
- 이메일
- 휴대폰 번호
- 신청 시각
- 결제 상태
- 참가 상태
- 승인된 회차 수
- 성공 여부
- 순위
- 액션

주요 액션:

- 결제 완료 처리
- 환불 완료 처리
- 참가 취소 처리

주의:

- 결제 완료 처리는 상태 변경 확인 모달을 띄운다.
- 상태 변경 후 감사 로그에 남는다는 안내를 작게 표시한다.

## 5.12 관리자 인증 심사

목적:

- 제출된 링크 인증을 빠르게 심사한다.

레이아웃:

- 좌측 또는 상단 필터
- 중앙 테이블
- 행 클릭 시 상세 패널 또는 모달

테이블 컬럼:

- 챌린지
- 참가자
- 회차
- 링크
- 설명 요약
- 제출 시각
- 상태
- 액션

상세 패널:

- 전체 링크
- 설명 전문
- 제출 정보
- 링크 열기
- 승인 버튼
- 반려 버튼

반려 모달:

- 반려 사유 선택
- 상세 사유 입력
- 반려 확정 버튼

UX:

- 링크 열기는 새 탭으로 동작한다.
- 승인/반려 후 현재 필터 상태를 유지한다.
- 승인/반려 결과는 토스트로 표시한다.

## 5.13 관리자 랭킹/성공자 화면

목적:

- 성공 여부와 순위를 확인하고 CSV를 다운로드한다.

상단:

- 챌린지 정보 요약
- 성공 필요 승인 수
- 재계산 버튼
- CSV 다운로드 버튼

테이블 컬럼:

- 순위
- 이름
- 이메일
- 승인된 회차 수
- 마지막 승인 시각
- 성공 여부
- 결제 상태
- 참가 상태

중요 UX:

- 재계산 버튼은 위험하지 않지만 결과를 바꾸므로 확인 모달을 둔다.
- CSV 다운로드는 감사 로그에 남는다는 문구를 표시한다.

## 5.14 감사 로그 화면

목적:

- 관리자 주요 행위를 추적한다.

필터:

- 수행자
- 행위 유형
- 기간

테이블 컬럼:

- 수행 시각
- 수행자
- 행위 유형
- 대상 유형
- 대상 ID
- 변경 전
- 변경 후
- IP 주소

디자인:

- 개발자/운영자용 화면이므로 정보 밀도를 높인다.
- JSON 값은 접기/펼치기 형태로 표현한다.

## 6. 디자인 시스템

## 6.1 컬러 토큰

추천 팔레트:

## 8. 인터랙션 규칙

## 8.1 저장/변경

- 저장 버튼 클릭 시 로딩 상태를 표시한다.
- 성공 시 토스트를 표시한다.
- 실패 시 필드 오류 또는 상단 오류를 표시한다.
- 상태 변경은 확인 모달을 거친다.

## 8.2 승인/반려

- 승인 버튼은 즉시 처리 가능하지만, 처리 후 토스트를 표시한다.
- 반려 버튼은 반드시 모달을 연다.
- 반려 사유 없이는 확정 버튼이 비활성화된다.

## 8.3 재계산

- 성공/랭킹 재계산은 확인 모달을 띄운다.
- 재계산 후 변경된 성공자 수와 갱신 시간을 표시한다.

## 8.4 CSV 다운로드

- CSV 다운로드 버튼 클릭 시 감사 로그에 남는다는 안내를 표시한다.
- 다운로드 실패 시 오류 토스트를 표시한다.

## 9. 반응형 기준

## 9.1 브레이크포인트

- 모바일: 0~767px
- 태블릿: 768~1023px
- 데스크톱: 1024px 이상

## 9.2 관리자 반응형

- 1024px 이상에서 최적화한다.
- 768~1023px에서는 사이드바를 접을 수 있게 한다.
- 767px 이하에서는 관리자 화면 사용은 가능하되 테이블은 가로 스크롤을 허용한다.

## 9.3 참가자 반응형

- 모바일 우선으로 구현한다.
- 주요 CTA는 엄지손가락 영역에 가깝게 배치한다.
- 긴 안내문은 접기/펼치기를 사용한다.
- 인증 제출 폼은 한 컬럼만 사용한다.

## 10. 접근성 기준

- 버튼과 링크는 키보드로 접근 가능해야 한다.
- 모든 입력 필드는 label을 가진다.
- 오류 메시지는 해당 필드와 연결한다.
- 색상만으로 상태를 구분하지 않는다.
- 배지에는 텍스트를 함께 표시한다.
- 모달은 ESC로 닫을 수 있다.
- 포커스 이동은 모달 안에 갇히도록 처리한다.
- 명도 대비는 일반 텍스트 기준 WCAG AA 수준을 목표로 한다.

## 11. 빈 상태, 오류 상태, 로딩 상태

## 11.1 빈 상태

챌린지 없음:

- “표시할 챌린지가 없습니다.”
- 관리자 화면에서는 `챌린지 만들기` 버튼 제공

인증 없음:

- “아직 제출된 인증이 없습니다.”

참가자 없음:

- “아직 참가 신청자가 없습니다.”

감사 로그 없음:

- “조건에 맞는 감사 로그가 없습니다.”

## 11.2 오류 상태

공통:

- “요청을 처리하지 못했습니다. 잠시 후 다시 시도해주세요.”

권한 없음:

- “접근 권한이 없습니다.”

검증 오류:

- 각 필드 아래에 구체적으로 표시

## 11.3 로딩 상태

- 전체 페이지 로딩은 스켈레톤보다 간단한 로딩 블록을 사용한다.
- 테이블 로딩은 행 스켈레톤을 사용한다.
- 버튼 로딩은 버튼 내부에 표시한다.

## 14. 프론트엔드 개발 체크리스트

- 권한별 라우팅이 분리되어 있는가?
- 참가자는 관리자 화면에 접근할 수 없는가?
- 관리자 전용 필드가 참가자 화면에 노출되지 않는가?
- 상태 배지가 모든 목록에 일관되게 적용되는가?
- 인증 제출이 모바일에서 불편하지 않은가?
- 반려 사유가 참가자에게 명확히 보이는가?
- CSV 다운로드와 상태 변경에 확인/피드백이 있는가?
- 빈 상태와 오류 상태가 준비되어 있는가?
- 테이블이 데스크톱에서 충분히 읽기 쉬운가?
- 긴 텍스트가 버튼이나 배지를 깨뜨리지 않는가?

## 15. 최종 디자인 판단

1차 프론트엔드는 “챌린지 운영을 멋있게 보여주는 서비스”가 아니라 “챌린지를 반복 운영하는 사람이 시간을 아끼는 도구”로 설계해야 한다.

관리자 화면은 테이블, 필터, 상태 배지, 빠른 액션 중심으로 만들고, 참가자 화면은 모바일에서 인증 제출과 상태 확인을 빠르게 끝내는 방향이 맞다.

디자인 차별점은 화려한 그래픽보다 상태 정보의 명확함, 심사 흐름의 속도, 청구 상태까지 함께 관리되는 운영 완성도에서 만든다.
