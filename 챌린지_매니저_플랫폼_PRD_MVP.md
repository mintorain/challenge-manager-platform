# 챌린지 매니저 플랫폼 MVP PRD

## 1. 문서 목적

이 문서는 기존 `챌린지 매니저 플랫폼 기획서`를 바탕으로 실제 개발을 시작할 수 있는 1차 MVP 범위를 정의한다.

기존 기획서는 장기적인 제품 방향과 확장 기능을 포함한다. 본 PRD는 첫 개발에서 구현할 기능, 제외할 기능, 화면, 데이터, 정책, 완료 기준을 명확히 하는 데 목적이 있다.

## 2. MVP 제품 정의

### 2.1 MVP 한 줄 정의

운영자가 블로그 챌린지를 만들고, 참가자가 링크와 사진으로 인증을 제출하면, 운영자가 승인/반려하고 성공자를 확인할 수 있는 챌린지 관리 플랫폼.

### 2.2 MVP 핵심 목표

- 실제 챌린지 1개를 운영할 수 있어야 한다.
- 참가자 인증 제출과 운영자 심사가 가능해야 한다.
- 챌린지 성공 여부를 시스템이 계산할 수 있어야 한다.
- 상금 지급 자동화는 하지 않더라도 지급 대상자 목록을 만들 수 있어야 한다.

### 2.3 MVP에서 검증할 가설

- 사용자는 챌린지에 참가하고 정해진 기간 동안 인증을 제출할 의사가 있다.
- 운영자는 링크와 사진 기반 인증을 수동 심사할 수 있다.
- 블로그 챌린지 하나만으로도 서비스의 핵심 운영 흐름을 검증할 수 있다.
- 자동 API 연동 없이도 초기 운영이 가능하다.

## 3. MVP 범위

### 3.1 포함 범위

- 관리자 로그인
- 참가자 회원가입 및 로그인
- 블로그 챌린지 생성
- 챌린지 기본 정보 수정
- 챌린지 상태 변경
- 챌린지 목록 및 상세 페이지
- 참가 신청
- 참가비 결제 상태 수동 관리
- 링크 및 사진 인증 제출
- 인증 승인/반려
- 반려 사유 입력
- 참가자별 인증 현황
- 챌린지 성공 여부 자동 계산
- 기본 랭킹
- 지급 대상자 목록
- 정산용 CSV 다운로드
- 관리자 감사 로그

### 3.2 제외 범위

- 실제 PG 결제 연동
- 자동 환불
- 자동 송금
- 유튜브 API 연동
- 틱톡 API 연동
- 블로그 본문 자동 크롤링
- 조회수 자동 수집
- 자동 원천징수 계산
- 카카오 알림톡
- 커뮤니티 피드
- 브랜드/스폰서 캠페인 리포트
- 모바일 앱
- AI 인증 심사
- 이메일 자동 발송
- 카카오/문자 자동 발송
- 관리자 다중 승인 워크플로우
- 계좌정보 저장
- 주민등록번호 또는 세금 신고용 고유식별정보 저장

### 3.3 MVP 이후 확장 예정

- 실제 결제 PG 연동
- 상금 자동 지급 또는 지급 API 연동
- 유튜브/틱톡 챌린지 타입 추가
- 외부 플랫폼 메타데이터 자동 수집
- 광고성 콘텐츠 표시 자동 검사
- 이의 제기 상세 워크플로우
- 세무 신고 자료 자동 생성

## 4. 사용자 유형

## 4.1 관리자

챌린지를 만들고 참가자, 인증, 성공 여부, 지급 대상자를 관리한다.

권한:

- 챌린지 생성/수정
- 참가자 목록 조회
- 참가자 결제 상태 변경
- 인증 승인/반려
- 성공 여부 재계산
- 정산용 CSV 다운로드
- 감사 로그 조회

## 4.2 참가자

챌린지에 참가하고 인증을 제출한다.

권한:

- 회원가입/로그인
- 챌린지 목록 조회
- 챌린지 상세 조회
- 참가 신청
- 내 인증 제출
- 내 인증 상태 확인
- 내 성공 여부 확인

## 5. 핵심 사용자 플로우

## 5.1 관리자 플로우

1. 관리자가 로그인한다.
2. 챌린지를 생성한다.
3. 모집 기간, 진행 기간, 참가비, 성공 기준을 입력한다.
4. 참가자 신청 현황을 확인한다.
5. 참가자의 결제 상태를 `결제 완료`로 변경한다.
6. 참가자가 제출한 인증을 확인한다.
7. 인증을 승인 또는 반려한다.
8. 챌린지 종료 후 성공 여부를 계산한다.
9. 성공자 및 지급 대상자 목록을 확인한다.
10. 정산용 CSV를 다운로드한다.

## 5.2 참가자 플로우

1. 참가자가 회원가입 또는 로그인한다.
2. 챌린지 목록에서 참여할 챌린지를 선택한다.
3. 챌린지 상세에서 기간, 참가비, 성공 기준을 확인한다.
4. 참가 신청을 한다.
5. 운영자가 결제 완료 처리하면 참가 확정 상태가 된다.
6. 진행 기간 동안 블로그 링크와 인증 사진을 제출한다.
7. 운영자의 승인/반려 결과를 확인한다.
8. 반려 시 사유를 확인하고 재제출한다.
9. 챌린지 종료 후 성공 여부와 순위를 확인한다.

## 6. MVP 정책

## 6.1 챌린지 유형 정책

MVP에서는 블로그 챌린지만 지원한다.

허용 링크:

- 네이버 블로그
- 티스토리
- 브런치
- 기타 운영자가 허용한 블로그 URL

MVP에서는 링크의 실제 본문 검사는 자동화하지 않는다. 운영자가 링크를 열어 수동 확인한다.

## 6.2 참가비 정책

MVP에서는 실제 결제 연동을 하지 않는다.

정책:

- 참가비 금액은 챌린지에 표시한다.
- 결제는 외부 계좌이체 또는 별도 방식으로 운영한다고 가정한다.
- 관리자가 참가자별 결제 상태를 수동으로 변경한다.
- 결제 완료 상태인 참가자만 인증 제출이 가능하다.

결제 상태:

- 결제 대기
- 결제 완료
- 참가 취소
- 환불 완료

## 6.3 성공 기준 정책

MVP 기본 성공 기준은 승인된 인증 개수 기준으로 한다.

기본값:

- 전체 미션 수: 운영자가 입력
- 성공 필요 승인 수: 운영자가 입력
- 예시: 7일 챌린지에서 5개 이상 승인 시 성공

성공 계산 규칙:

- `승인된 인증 수 >= 성공 필요 승인 수`이면 성공
- 반려, 검토 중, 취소된 인증은 성공 계산에서 제외
- 동일 미션 회차에 여러 인증이 있을 경우 승인된 최신 인증 1개만 인정

## 6.4 랭킹 정책

MVP 랭킹은 단순하고 설명 가능한 기준으로 한다.

정렬 기준:

1. 승인된 인증 수가 많은 순
2. 마지막 승인 시간이 빠른 순
3. 참가 신청 시간이 빠른 순

MVP에서는 조회수, 좋아요 수, 댓글 수를 랭킹에 반영하지 않는다.

## 6.5 상금 및 지급 대상자 정책

MVP에서는 자동 상금 지급을 하지 않는다.

지원 범위:

- 성공자 목록 표시
- 지급 대상자 목록 표시
- 관리자 입력용 지급 예정 금액 필드
- CSV 다운로드

제외 범위:

- 자동 상금 계산
- 원천징수 자동 계산
- 자동 송금
- 지급명세서 자동 생성

운영 원칙:

- 실제 상금 계산과 세무 처리는 운영자가 외부에서 확정한다.
- 플랫폼은 지급 대상자와 증빙 데이터를 제공하는 수준으로 제한한다.

## 6.6 인증 제출 정책

인증 제출 필수값:

- 미션 회차
- 블로그 링크
- 인증 설명

선택값:

- 인증 사진

제출 제한:

- 참가 확정 상태에서만 제출 가능
- 챌린지 진행 기간 내에만 제출 가능
- 동일 회차에 재제출 가능
- 재제출 시 기존 인증은 이력으로 남긴다.

## 6.7 반려 및 재제출 정책

반려 사유는 필수 입력으로 한다.

기본 반려 사유:

- 링크 접속 불가
- 기간 외 콘텐츠
- 필수 조건 미충족
- 중복 제출
- 부적절한 콘텐츠
- 기타

재제출:

- 반려된 회차는 진행 기간 내 재제출 가능
- 재제출된 인증은 새 인증 레코드로 저장
- 기존 반려 이력은 삭제하지 않음

## 6.8 상태 전이 정책

상태 변경은 정해진 흐름만 허용한다. 잘못된 상태 전이를 막아야 정산, 랭킹, 감사 로그가 일관성을 유지한다.

### 6.8.1 챌린지 상태 전이

허용 전이:

- `임시 저장 -> 모집 중`
- `모집 중 -> 진행 중`
- `진행 중 -> 종료`
- `종료 -> 정산 완료`
- `모집 중 -> 임시 저장`

제한:

- `정산 완료` 상태의 챌린지는 수정할 수 없다.
- `진행 중` 상태에서는 성공 기준, 참가비, 진행 기간을 수정할 수 없다.
- `종료` 이후에는 인증 제출을 받을 수 없다.
- 관리자만 상태를 변경할 수 있다.

### 6.8.2 참가 상태 전이

허용 전이:

- `신청 -> 참가 확정`
- `신청 -> 참가 취소`
- `참가 확정 -> 참가 취소`
- `참가 확정 -> 성공`
- `참가 확정 -> 실패`

제한:

- `참가 취소` 상태는 인증 제출이 불가능하다.
- `성공` 또는 `실패` 상태는 성공 여부 재계산 시에만 변경한다.
- 참가자가 직접 참가 상태를 변경할 수 없다.

### 6.8.3 결제 상태 전이

허용 전이:

- `결제 대기 -> 결제 완료`
- `결제 대기 -> 참가 취소`
- `결제 완료 -> 환불 완료`
- `결제 완료 -> 참가 취소`

제한:

- MVP에서는 실제 결제 취소 API를 호출하지 않는다.
- 결제 상태 변경은 관리자만 가능하다.
- 결제 상태 변경 시 감사 로그를 남긴다.

### 6.8.4 인증 상태 전이

허용 전이:

- `검토 중 -> 승인`
- `검토 중 -> 반려`
- `승인 -> 반려`
- `반려 -> 취소`

제한:

- 참가자는 제출 후 인증 상태를 직접 변경할 수 없다.
- 승인된 인증을 반려로 바꾸는 경우 사유 입력이 필수다.
- 상태 변경 이력은 `SubmissionHistory`와 `AuditLog`에 모두 기록한다.

## 6.9 입력값 검증 정책

### 6.9.1 챌린지 생성 검증

- 챌린지명은 필수이며 2자 이상 80자 이하로 제한한다.
- 설명은 필수이며 20자 이상 입력해야 한다.
- 모집 시작일은 모집 종료일보다 이전이어야 한다.
- 모집 종료일은 진행 시작일보다 늦을 수 없다.
- 진행 시작일은 진행 종료일보다 이전이어야 한다.
- 참가비는 0원 이상 숫자만 허용한다.
- 전체 미션 수는 1 이상 100 이하로 제한한다.
- 성공 필요 승인 수는 1 이상 전체 미션 수 이하로 제한한다.
- 최대 참가자 수는 1 이상 10,000 이하로 제한한다.

### 6.9.2 참가 신청 검증

- 모집 중 상태의 챌린지만 참가 신청할 수 있다.
- 동일 사용자는 같은 챌린지에 중복 신청할 수 없다.
- 최대 참가자 수에 도달한 챌린지는 참가 신청할 수 없다.
- 참가 신청 시 약관 동의와 개인정보 수집 동의가 필수다.

### 6.9.3 인증 제출 검증

- 참가 확정 상태에서만 인증 제출이 가능하다.
- 챌린지 진행 기간 내에만 인증 제출이 가능하다.
- 미션 회차는 1 이상 전체 미션 수 이하만 허용한다.
- 링크 URL은 `http://` 또는 `https://`로 시작해야 한다.
- 링크 URL은 2,000자 이하로 제한한다.
- 인증 설명은 10자 이상 1,000자 이하로 제한한다.
- 인증 사진은 선택값이지만 업로드 시 허용 확장자와 용량 제한을 적용한다.

## 6.10 알림 정책

MVP에서는 외부 알림을 자동 발송하지 않는다.

지원 범위:

- 화면 안에서 상태 변경 결과를 표시한다.
- 참가자는 내 인증 내역 화면에서 승인/반려 여부를 확인한다.
- 참가자는 반려 사유를 인증 내역 화면에서 확인한다.

제외 범위:

- 이메일 알림
- 문자 알림
- 카카오 알림톡
- 앱 푸시 알림

후속 버전에서는 인증 반려, 마감 임박, 성공 확정, 지급 대상 확정 알림을 추가한다.

## 7. 화면 요구사항

## 7.1 공통 화면

### 7.1.1 로그인

기능:

- 이메일 로그인
- 비밀번호 로그인
- 관리자/참가자 권한 분기

필드:

- 이메일
- 비밀번호

완료 기준:

- 로그인 성공 시 권한에 맞는 대시보드로 이동한다.
- 로그인 실패 시 오류 메시지를 표시한다.

### 7.1.2 회원가입

기능:

- 참가자 회원가입

필드:

- 이름
- 이메일
- 비밀번호
- 휴대폰 번호
- 약관 동의
- 개인정보 수집 동의

완료 기준:

- 필수값 입력 후 계정 생성이 가능하다.
- 이메일 중복 가입을 막는다.

## 7.2 참가자 화면

### 7.2.1 챌린지 목록

기능:

- 모집 중 챌린지 목록 조회
- 진행 예정 챌린지 조회
- 진행 중 챌린지 조회

## 10. 성공 계산 로직

## 10.1 승인 수 계산

참가자별로 미션 회차 기준 승인 인증을 계산한다.

규칙:

- 같은 회차에 승인 인증이 여러 개 있으면 가장 최근 승인 1개만 인정한다.
- 반려된 인증은 제외한다.
- 검토 중 인증은 제외한다.

## 10.2 성공 판정

```
if approved_mission_count >= challenge.required_approval_count:
    success_status = "success"
else:
    success_status = "failed"
```

## 10.3 랭킹 계산

정렬:

1. `approved_mission_count` 내림차순
2. `last_approved_at` 오름차순
3. `joined_at` 오름차순

## 10.4 성공 계산 예외 처리

- 참가 취소 상태인 참가자는 성공 계산에서 제외한다.
- 결제 완료가 아닌 참가자는 성공 계산에서 제외한다.
- 챌린지 종료 전에는 성공 상태를 `in_progress`로 표시한다.
- 챌린지 종료 후 재계산을 실행하면 `success` 또는 `failed`로 확정한다.
- 관리자가 승인 상태를 수정한 경우 성공 여부와 랭킹을 다시 계산해야 한다.
- 성공 여부 재계산 시 기존 랭킹과 성공 상태를 덮어쓰기 전에 감사 로그를 남긴다.

## 10.5 CSV 생성 규칙

- CSV는 관리자만 다운로드할 수 있다.
- CSV 다운로드 시 감사 로그를 남긴다.
- CSV 파일명은 `challenge-{challenge_id}-payouts-{YYYYMMDD}.csv` 형식을 사용한다.
- CSV에는 계좌번호, 주민등록번호, 고유식별정보를 포함하지 않는다.
- 한글 깨짐을 줄이기 위해 UTF-8 BOM 포함 CSV를 권장한다.

## 11. 비기능 요구사항

## 11.1 보안

- 비밀번호는 해시로 저장한다.
- 관리자 API는 관리자 권한이 있어야 접근 가능하다.
- 인증 제출 파일은 허용 확장자와 최대 용량을 제한한다.
- CSV 다운로드는 관리자만 가능하다.
- 주요 관리자 행위는 감사 로그에 기록한다.

## 11.2 파일 업로드

허용 파일:

- jpg
- jpeg
- png
- webp

최대 크기:

- 파일 1개당 5MB

MVP 제한:

- 인증 사진은 제출 1건당 1장만 허용한다.

## 11.3 성능

MVP 기준:

- 챌린지 100개
- 챌린지당 참가자 1,000명
- 참가자당 인증 30건

목표:

- 목록 화면 응답 2초 이내
- 인증 심사 상태 변경 1초 이내
- CSV 다운로드 10초 이내

## 11.4 접근성 및 사용성

- 모바일 브라우저에서 참가자 인증 제출이 가능해야 한다.
- 관리자 화면은 데스크톱 사용을 우선한다.
- 반려 사유와 성공 기준은 사용자가 이해할 수 있는 문장으로 표시한다.

## 11.5 개인정보 최소화

MVP에서는 상금 지급 자동화와 세무 신고 자동화를 하지 않으므로 민감한 개인정보를 수집하지 않는다.

수집 항목:

- 이름
- 이메일
- 휴대폰 번호
- 서비스 이용 기록
- 인증 제출 링크
- 인증 사진

수집하지 않는 항목:

- 계좌번호
- 주민등록번호
- 신분증 사본
- 세금 신고용 고유식별정보

운영 원칙:

- 정산 CSV에는 지급 대상자 식별에 필요한 최소 정보만 포함한다.
- 계좌정보가 필요한 경우 MVP 외부 운영 프로세스에서 별도로 관리한다.
- 후속 버전에서 계좌정보를 저장하려면 암호화, 접근권한 분리, 열람 로그를 먼저 설계한다.

## 11.6 장애 및 운영 예외

- 파일 업로드 실패 시 인증 제출은 저장하지 않고 사용자에게 재시도를 안내한다.
- 인증 승인/반려 처리 중 오류가 발생하면 상태를 변경하지 않는다.
- 성공 여부 재계산 중 오류가 발생하면 기존 성공 상태와 랭킹을 유지한다.
- CSV 생성 중 오류가 발생하면 다운로드 이력을 성공으로 남기지 않는다.
- 관리자 작업 실패는 사용자 데이터가 중간 상태로 남지 않도록 트랜잭션으로 처리한다.

## 12. 수용 기준

## 12.1 관리자 기준

- 관리자는 챌린지를 생성할 수 있다.
- 관리자는 챌린지 상태를 변경할 수 있다.
- 관리자는 참가자의 결제 상태를 변경할 수 있다.
- 관리자는 제출 인증을 승인할 수 있다.
- 관리자는 제출 인증을 반려하고 사유를 남길 수 있다.
- 관리자는 성공자 목록을 확인할 수 있다.
- 관리자는 정산용 CSV를 다운로드할 수 있다.
- 관리자의 주요 행위가 감사 로그에 남는다.

## 12.2 참가자 기준

- 참가자는 회원가입과 로그인을 할 수 있다.
- 참가자는 모집 중 챌린지를 볼 수 있다.
- 참가자는 챌린지에 참가 신청할 수 있다.
- 참가자는 참가 확정 후 인증을 제출할 수 있다.
- 참가자는 내 인증 상태를 확인할 수 있다.
- 참가자는 반려 사유를 확인할 수 있다.
- 참가자는 내 성공 여부를 확인할 수 있다.

## 12.3 시스템 기준

- 시스템은 승인된 인증 수를 기준으로 성공 여부를 계산한다.
- 시스템은 정의된 정렬 기준으로 랭킹을 계산한다.
- 시스템은 같은 회차의 중복 인증을 성공 계산에서 1개만 인정한다.
- 시스템은 검토 중 인증을 성공 계산에서 제외한다.

## 12.4 예외 케이스 기준

- 모집 기간이 종료된 챌린지는 참가 신청 버튼이 비활성화된다.
- 최대 참가자 수에 도달한 챌린지는 참가 신청이 불가능하다.
- 결제 완료가 아닌 참가자는 인증 제출이 불가능하다.
- 진행 기간이 아닌 챌린지는 인증 제출이 불가능하다.
- 반려 처리 시 반려 사유가 없으면 저장할 수 없다.
- 승인된 인증을 반려로 변경하면 성공 여부 재계산 대상이 된다.
- 정산 완료 상태의 챌린지는 인증 심사와 참가자 상태 변경이 제한된다.

## 13. 개발 우선순위

## 13.1 1차 스프린트

- 프로젝트 기본 구조
- 사용자 로그인/회원가입
- 관리자 권한 분기
- 챌린지 생성/목록/상세

## 13.2 2차 스프린트

- 참가 신청
- 참가자 관리
- 결제 상태 수동 변경
- 내 챌린지 화면

## 13.3 3차 스프린트

- 인증 제출
- 파일 업로드
- 인증 내역
- 관리자 인증 심사

## 13.4 4차 스프린트

- 성공 여부 계산
- 랭킹
- 성공자 목록
- 정산 CSV 다운로드
- 감사 로그

## 14. 개발 전 확정 필요 항목

아래 항목은 개발 시작 전에 최종 확정해야 한다.

- 사용할 기술 스택
- 실제 로그인 방식: 이메일/비밀번호 또는 소셜 로그인
- 파일 저장 위치: 로컬, S3, Supabase Storage 등
- 관리자 계정 생성 방식
- 참가비 결제 처리 방식
- 챌린지 환불 안내 문구
- 기본 성공 기준
- 정산 CSV에 포함할 개인정보 범위
- 배포 환경

## 15. MVP 권장 기본 결정안

아래 결정안은 개발 시작 전 별도 지시가 없을 경우 기본값으로 사용한다.

- 서비스 형태: 관리자 화면과 참가자 화면을 하나의 웹 서비스 안에서 권한으로 분기
- 첫 챌린지 타입: 블로그 챌린지
- 로그인 방식: 이메일/비밀번호
- 결제 방식: 실제 PG 미연동, 관리자 수동 결제 상태 변경
- 상금 지급 방식: 자동 지급 없음, 지급 대상자 CSV 제공
- 계좌정보: 시스템에 저장하지 않음
- 성공 기준: 승인 인증 수 기준
- 랭킹 기준: 승인 인증 수, 마지막 승인 시간, 참가 신청 시간
- 알림 방식: MVP에서는 화면 내 상태 표시만 제공
- 관리자 계정: 초기 seed 계정 또는 수동 DB 등록 방식
- 파일 저장: 개발 환경에서는 로컬 저장, 배포 환경에서는 별도 스토리지 확정 필요

## 16. 최종 MVP 판단

첫 버전은 “상금 지급 자동화 플랫폼”이 아니라 “챌린지 운영과 성공자 산출을 관리하는 플랫폼”으로 출시하는 것이 현실적이다.

이렇게 범위를 제한하면 법무, 세무, 결제, 외부 API 리스크를 줄이면서도 서비스의 핵심 가치를 검증할 수 있다. 이후 실제 결제, 자동 지급, 유튜브/틱톡 자동 검증은 운영 데이터가 쌓인 뒤 확장한다.
